LLM Council Report · June 5, 2026

Is preliminary design → OpenRocket → electronics the right starting sequence for this rocket project?

5 advisors · 4 peer reviews · 1 chairman synthesis
The One Thing to Do First

Write the terminal phase simulation script today — a single Python file, under 100 lines — that takes descent velocity, detection altitude, and a latency budget (sensor + compute + igniter + spool) as inputs and outputs the residual ignition window under worst-case assumptions. Do not open OpenRocket. Do not finalize the BOM. Do not design the gimbal. That number either unlocks the project or forces a redesign, and you need to know which one it is before June ends.

Chairman's Verdict

Where the Council Agrees

All five advisors converge on the same core verdict: the terminal phase simulation must happen before anything else, and the proposed sequence is wrong for burying it. The agreement is not just sequencing preference — it is unanimous that this single script is the load-bearing calculation that either validates or kills the architecture. Every downstream decision (sensor selection, polling rate, state machine thresholds, leg deployment timing) has a hard dependency on that number.

The council also agrees that the TVC gimbal is the true critical-path item and that mechanical iteration must begin no later than week 2, because first-time 3D-printed gimbals require multiple revision cycles and PID tuning is not a two-week task.

Where the Council Clashes

The one genuine tension is between The Expansionist and the rest of the council on framing. The Expansionist treats the ignition window as a solved constraint and pivots to upside: parallelizing OpenRocket with electronics bring-up, publishable data, conference-quality work. The other four treat the window as an open feasibility question that must be answered before any of that matters. A builder reading the Expansionist might treat the sim as productive enrichment. A builder reading the others understands it is a go/no-go before a single part ships. The peer reviews called this out four times independently.

There is also a secondary tension on parallelism. The Expansionist and the Outsider recommend running OpenRocket in parallel with electronics bring-up. The First Principles Thinker implicitly disagrees — OpenRocket is downstream of the terminal phase sim because the descent profile is an input, not an output, of OpenRocket for this use case. The Outsider is correct that gimbal mechanical build has zero dependencies on the sim and can run in parallel with it. OpenRocket cannot.

Blind Spots the Council Caught

F-class motor ignition jitter is not a mean-value problem. The simulation can model expected ignition delay, but Estes motors have a tail distribution on ignition latency across units. A unit that fires 80ms late eats a third of the best-case window. The architecture needs to be stress-tested against worst-case ignition delay from a real sample distribution, not just the nominal spec.

Motor reliability under flight conditions was never questioned. Vibration, G-loading, and potential moisture affect pass rate. A motor that lights reliably on a bench is not guaranteed to light reliably under the dynamic conditions of a terminal descent.

Motor availability and lead time. Estes F-class motors are hobby-retail items with inconsistent stock. Order motors on day one, before the simulation even finishes running.

The Recommendation

The proposed sequence is wrong and should be discarded. The correct sequence:

1. Terminal phase sim — day one. Run with worst-case latency values stacked: BMP388 at its slowest oversampling setting, Teensy interrupt-to-output latency, igniter delay at the 90th percentile, motor spool time. If the residual window is positive with margin, proceed. If it is negative or inside noise, the sensor stack, the detection altitude, or the motor selection must change before a single component is ordered.

2. Order motors immediately — in parallel with writing the simulation. Stock risk is real and silent.

3. TVC gimbal physical build begins week 2, not after electronics integration. It runs in parallel with everything else — no upstream dependencies except gimbal geometry, already constrained by the motor mount.

4. OpenRocket runs after the terminal phase sim confirms the descent profile, because the ascent model must be consistent with what the landing motor must deliver.

5. Electronics integration (IMU, barometer, flight computer) comes after the sim validates polling rate and state machine timing requirements — those requirements flow from the sim output.

Full Advisor Responses

The Contrarian Agrees with consensus
The proposed sequence is backwards in a way that matters. The builder lists "preliminary design + component list" as step one, but that's not actually first — it's already been done. The architecture is locked: Teensy 4.0, BMP388, F-class motors, TVC gimbal, spring legs. The sequence being debated is a fiction. The real question is what to validate now before the hardware assumptions harden into sunk cost.

The BMP388 at standard configuration has 40–180ms response latency depending on oversampling settings. That alone can eat half the margin before you account for IMU fusion, state estimation, and ignition circuit delay. The critical path risk isn't TVC gimbal slipping past week 6. It's discovering in week 7 that the ignition timing assumption was wrong and the sensor stack can't close the loop in time. That's not a schedule problem. That's a redesign.

Run the terminal phase sim first. Today. Before touching OpenRocket.
The First Principles Thinker Strongest (4/4 reviewers)
The 200–400ms ignition window is not a design preference — it is a physical constraint derived from kinematics. At 12 m/s descent and 3m altitude, you have roughly 250ms of total time before ground contact. That number, minus sensor latency, minus compute latency, minus igniter delay, minus motor spool time, determines whether this project is physically possible at all with Estes F-class motors.

The BMP388 has a configurable ODR; at 50Hz you get 20ms per sample. The Teensy 4.0 interrupt-to-output latency is measurable but non-zero. Estes igniters have statistical ignition delay distributions, not point values. F-class motors have non-negligible spool time before meaningful thrust. Stack those numbers. If the residual window is negative or inside measurement noise, the architecture is dead on first principles — no amount of TVC tuning rescues it.

Correct sequence: kinematic feasibility calculation → latency budget derivation → component selection constrained by that budget → OpenRocket → hardware build.
The Expansionist Biggest blind spot (4/4 reviewers)
The OpenRocket step is fine but it should run in parallel with electronics bring-up, not sequentially before it. Gimbal fabrication and PID loop development on a test stand can start as early as week 2. The week-6 TVC deadline is only safe if mechanical iteration starts immediately — 3D-printed gimbals almost always need 2–3 revision cycles.

The biggest opportunity: this project generates genuinely publishable data on a shoestring budget. If the simulation is rigorous and the flight log is clean, this is conference-quality work for a high schooler. That only happens if the sim is the foundation, not an afterthought dressed up as step one.

Note: Peer reviewers flagged that framing the sim as an "opportunity" softens what should be a hard go/no-go gate. Also, OpenRocket outputs depend on the terminal phase sim — they cannot truly run in parallel.
The Outsider Agrees with consensus
The proposed sequence has a classic waterfall trap baked into it: three sequential phases before you touch real hardware, with the critical-path item (TVC gimbal + PID) sitting at the end of a dependency chain. Every week spent on OpenRocket modeling is a week you're not discovering that your gimbal servo has 40ms latency, or that your Teensy's interrupt handling hiccups under sensor fusion load.

The terminal phase simulation script is your interface contract. In distributed systems, you define the API before you build the services. That script defines the timing budget, which defines the sensor polling rate, which defines the firmware architecture, which defines the gimbal response spec. Everything downstream is an implementation of that contract.

Concrete recommendation: parallel-path the gimbal mechanical build with the simulation script. They have zero dependencies on each other and both feed into PID tuning. Running them sequentially burns two weeks you don't have.
The Executor Agrees with consensus
The terminal phase simulation must be the literal first commit, not a line item in step 1. Every downstream decision has a hard dependency on that number. Motor selection, leg deployment timing, altimeter polling rate, the state machine transition threshold — all of it shifts if the window turns out to be 150ms instead of 350ms.

The correct sequence: (1) Terminal phase sim — day one, Python, one file, 50 lines. (2) OpenRocket ascent modeling, informed by the landing motor constraints. (3) TVC gimbal physical build starts no later than week 2.

The critical path warning is underweighted. TVC gimbal plus PID tuning is not a two-week task for a first-time builder. Mechanical slop, motor mount tolerances, and PID oscillation each add days. Week 6 as the hard deadline means the gimbal must be spinning and attitude-stabilizing by week 5 to have any buffer. Build the sim today. Start cutting gimbal parts this week.
